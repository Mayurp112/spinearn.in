Request flow — developer sees an ad

1
Extension polls GET /api/v1/ads/current
Every 30–60 s when an AI spinner is active. Nginx forwards to FastAPI. Returns impression_id, creative_text, click_url.
2
Auction picks winner from Redis ZSET
Campaigns stored as sorted set scored by bid_cpm. Highest bidder with remaining impressions wins. Cache TTL 5000 ms.
3
Extension shows ad, starts 5 s timer
ImpressionTracker ticks every 100 ms. Only counts when VS Code window is focused. At 5 s calls POST /api/v1/metrics/impression/confirm.
4
Backend credits developer balance
Fraud caps checked (50¢/hour, $5/day). developer_earn added to pending_balance. Campaign impressions_served incremented.
5
Developer requests payout → Celery task dispatched
India → Razorpay Route (bank/UPI). Global → Wise (IBAN/SWIFT/ACH). Minimum $10 USD. Payout confirmed via webhook (Razorpay) or immediate fund call (Wise).
Deployment
Current Plan — 0 to 1,000 developers
Single EC2 instance, all services in Docker Compose. Verified to handle 1k developers at <5% backend capacity.
Single EC2 t3.large · 8 GB RAM · 2 vCPU
₹4,000–6,500 / month
nginx fastapi + uvicorn postgres 16 redis 7 celery worker celery beat next.js frontend
Memory used at baseline: ~2.8 GB of 8 GB (35%). Peak backend load at 1k users: ~20 req/s vs 800 req/s capacity.
On-demand cost: t3.large ≈ ₹6,500/month. 1-year reserved: ≈ ₹4,000/month.

3 fixes required before production

Critical
Remove --reload

Found in docker-compose.yml uvicorn command. Dev-only flag. Scans filesystem continuously. Switch to --workers 2.

High
PostgreSQL backups

No backup job exists. Add nightly pg_dump cron to S3 or a second volume. Developer balances are real money.

Medium
Server-side impression check

5 s timer is client-side only. Add server validation: confirmed_at - created_at >= 4.5 s. Prevents fake confirms.

Deployment
Scale Plan — 2,000 to 5,000 developers
At 2k+ the first bottleneck is PostgreSQL connection exhaustion and disk I/O contention on a single machine. The solution is separating stateful services onto managed infrastructure.
Tier 2 — 2,000 developers
₹14,000–18,000 / month
EC2 t3.large (app only) RDS db.t3.medium (postgres) ElastiCache t3.micro (redis)
Move PostgreSQL to Amazon RDS — managed backups, automatic failover, separate IOPS. Move Redis to ElastiCache — dedicated memory, no contention with app containers. EC2 now runs only nginx + FastAPI + Celery + Next.js.

At 2k devs peak load reaches ~40 req/s. DB connection pool (30) is sufficient because async SQLAlchemy releases connections immediately. Add PgBouncer if you see connection wait times.
Tier 3 — 5,000 developers
₹32,000–42,000 / month
2x EC2 t3.large behind ALB RDS db.t3.large + read replica ElastiCache t3.small Application Load Balancer
Peak load at 5k devs: ~80–100 req/s. Add ALB in front of two EC2s (active-active). Read replica for GET-heavy routes (earnings, leaderboard). Next.js can move to Vercel free tier — removes frontend load from EC2 entirely.

What breaks first (in order)

#BottleneckTriggers atFix
1PostgreSQL connections exhausted~1,500 devsRDS + PgBouncer
2EC2 disk I/O (Postgres + Redis competing)~1,500 devsSeparate to RDS + ElastiCache

Extension
VS Code Marketplace Publishing
Publishing to the Visual Studio Marketplace makes the extension installable worldwide. Publisher ID in package.json: spinads. Version: 1.0.0.

One-time setup

1
Create publisher account
Go to marketplace.visualstudio.com/manage. Create publisher with ID spinads. Requires a Microsoft account.
2
Generate Personal Access Token (PAT)
Azure DevOps (dev.azure.com) → User Settings → Personal Access Tokens. Scope: Marketplace → Manage. Save the token — shown only once.
3
Login with vsce
npx vsce login spinads
# Enter your PAT when prompted
4
Add required marketplace fields to package.json
"icon": "assets/icon.png",  // 128x128 PNG required
"repository": { "type": "git", "url": "https://github.com/yourorg/spinads" },
"license": "MIT",
"homepage": "https://spinads.dev"

Every release — commands

# Build + type check
npm run type-check && npm run build

# Package into .vsix (inspect before publishing)
npx vsce package

# Publish directly
npx vsce publish

# Or publish a specific .vsix
npx vsce publish --packagePath spinads-1.2.0.vsix

Auto-update behaviour

VS Code checks the marketplace on every startup. If a newer version exists it downloads and installs silently in the background, then prompts the user to reload the window. No manual distribution needed once published.

Extension
Versioning Strategy
Current version: 1.0.0. Follows semantic versioning — MAJOR.MINOR.PATCH. The version field in extension/package.json is the single source of truth.
MAJOR · X.0.0

Breaking changes. New minimum VS Code version. Auth method change that logs all users out. Removes a command or setting users rely on.

npx vsce publish major
MINOR · 1.X.0

New features: new adapter (Cursor, Windsurf), new commands, new status bar behaviour. Backwards compatible.

npx vsce publish minor
PATCH · 1.0.X

Bug fixes, performance improvements, dependency bumps. No new behaviour. Most common release type.

npx vsce publish patch

Version bump workflow

# Option A — vsce bumps automatically
npx vsce publish patch    # edits package.json, publishes, done

# Option B — manual (more control, creates git tag)
npm version patch         # edits package.json + creates git tag
npx vsce package          # inspect .vsix before publishing
npx vsce publish
git push --tags

Remote kill switch + min_version enforcement

Your extension already polls GET /api/v1/config for global_kill_switch. Add a min_version field to this response — the extension compares it against its own version and shows a "please update" notification if behind. VS Code auto-updates in the background, but the notification accelerates it for critical fixes.
